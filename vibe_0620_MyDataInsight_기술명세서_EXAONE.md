# MyDataInsight App 기술명세서 (TRD)
> EXAONE3.5:7.8b 로컬 LLM 기반 CSV 데이터 분석 웹 애플리케이션
> 원본(GPT API 버전) 대비 변경점은 본문에 표시

## 0. 변경 개요

원본 기획서(GPT-4 / OpenAI API 기반)를 **EXAONE3.5:7.8b(Ollama 로컬 구동)** 기준으로 전환한 명세서입니다.

| 구분 | 원본 | 변경 |
|---|---|---|
| AI 모델 | OpenAI GPT-4 / GPT-4o (외부 API) | EXAONE3.5:7.8b (Ollama, 로컬) |
| 호출 방식 | `https://api.openai.com/v1/chat/completions` | `http://localhost:11434/api/chat` |
| 인증 | OpenAI API Key 필요 | 불필요 (로컬 실행) |
| 데이터 외부 전송 | GPT API로 샘플 데이터 전송 | **없음** (전 과정 로컬 처리) |
| 비용 | 토큰 사용량 기반 과금 | 없음 (로컬 컴퓨팅 자원만 사용) |
| 네트워크 의존성 | 인터넷 연결 필수 | 불필요 (오프라인 동작 가능) |

> ⚠️ 확인 필요: EXAONE3.5:7.8b의 응답 품질·속도는 GPT-4 계열과 다를 수 있으며, 실제 비교는 구현 후 테스트가 필요합니다(추정하지 않음).

## 1. 프로젝트 개요

**MyDataInsight**는 사용자가 업로드한 CSV 파일을 자동으로 분석하고, **EXAONE 로컬 LLM**을 활용하여 인사이트를 도출하며, 시각화 차트를 제공하는 React 기반 웹 애플리케이션입니다.

### 핵심 가치
- 간편한 데이터 분석: CSV 파일만 업로드하면 즉시 분석
- **로컬 AI 기반 인사이트**: 외부 API 없이 EXAONE이 데이터 패턴·트렌드·이상치 분석
- 직관적 시각화: 자동 생성된 차트로 데이터 이해도 향상
- **데이터 보안**: 모든 처리가 로컬에서 이루어져 외부 전송 없음 (공공/민감 데이터에 강점)

## 2. 전체 사용자 플로우

```
CSV 업로드
   ↓
데이터 미리보기 & 컬럼 선택 (상위 N행, 타입 추론, 체크박스)
   ↓
자동 EDA 실행 (요약 통계, 결측치, 분포)
   ↓
시각화 대시보드 (히스토그램/바차트/산점도/상관계수)
   ↓
인사이트 요약 (EXAONE 연동)
   ↓
결과 내보내기 (Markdown/텍스트 다운로드)
```

## 3. 시스템 아키텍처

### 3.1 전체 구조

원본 기획(클라이언트 사이드 전용)과 달리, **로컬 LLM(EXAONE/Ollama)은 브라우저에서 직접 호출이 불가능**하므로 최소한의 백엔드(프록시 서버)가 필요합니다.

```
[사용자]
   ↓ CSV 업로드
[React Frontend]
   ↓ File 객체
[Papaparse] → JSON 변환 (클라이언트 처리)
   ↓
[백엔드 API] (Node/Express 또는 Python/FastAPI)
   ↓ 통계/상관분석 계산 (`/api/analyze`)
   ↓ HTTP 요청
[Ollama 로컬 서버] (localhost:11434, EXAONE3.5:7.8b)
   ↓ 분석 결과
[결과 표시 + 차트 렌더링]
```

> 📌 핵심 차이: 원본은 "완전 클라이언트 사이드(서버 업로드 없음)"를 표방했으나, Ollama는 브라우저에서 직접 호출 시 CORS 제약이 있어 **경량 백엔드(프록시) 경유가 일반적**입니다. (이전 주간보고 자동화 앱에서 사용한 구조와 동일)
> 📌 통계 계산 위치 통일: 어차피 EXAONE 호출용 백엔드 프록시가 필요한 구조이므로, 통계/상관분석(`/api/analyze`)도 클라이언트가 아닌 **백엔드에서 일괄 계산**하는 것으로 통일함(섹션 3.3 API 표와 일치). 클라이언트는 파싱·렌더링만 담당.

### 3.2 프론트엔드 화면/컴포넌트 구조

| 컴포넌트 | 역할 |
|---|---|
| `UploadView` | CSV 파일 업로드, 상태 표시 |
| `DataPreview` | 테이블 그리드(상위 N행), 컬럼 타입 표시, 컬럼 선택 체크박스 |
| `AnalysisDashboard` | 탭 구조: 기본통계 / 시각화 / 상관분석 |
| `InsightPanel` | EDA 결과 텍스트 요약 + "EXAONE에게 인사이트 설명 요청" 버튼 + 응답 카드 |
| `SettingsPanel` | 샘플 행 개수, 차트 옵션, 언어/포맷 옵션 |

### 3.3 백엔드 / API 기능 구조

| API | 메서드 | 설명 | AI 연동 여부 |
|---|---|---|---|
| `/api/upload` | POST | CSV FormData 수신, 파싱, 메타정보(열 이름·타입·행 수) 반환 | 무관 |
| `/api/analyze` | POST | 선택 컬럼 기준 요약 통계·결측치·상관계수 계산 후 JSON 반환 | 무관 |
| `/api/plot-data` | POST | 차트 유형별 데이터 가공 반환 | 무관 |
| `/api/insights` | POST | EDA 결과를 EXAONE에 전달, 자연어 인사이트 생성 후 반환 | **EXAONE 호출** |

> 📌 네이밍 참고: 강사 원본(3.4강 실습 워크북)은 이 엔드포인트를 `/api/gpt`로 표기함. 본 프로젝트는 모델 종속성을 드러내지 않기 위해 `/api/insights`로 명명하며, 기능적으로는 동일한 위치(유일한 AI 호출 지점)임. Cursor가 강사 노션 기획서를 추가 참조할 경우 두 네이밍이 혼재될 수 있으므로, 구현 시 `/api/insights`로 통일할 것.

#### `/api/insights` 상세 (원본 대비 유일한 변경 지점)

**원본 (GPT)**
```javascript
const response = await fetch('https://api.openai.com/v1/chat/completions', {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${OPENAI_API_KEY}`,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    model: 'gpt-4o',
    messages: [{ role: 'user', content: prompt }],
    max_tokens: 1000
  })
});
```

**변경 (EXAONE / Ollama)**
```javascript
const response = await fetch('http://localhost:11434/api/chat', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    model: 'exaone3.5:7.8b',
    messages: [{ role: 'user', content: prompt }],
    stream: false
  })
});
```

> ⚠️ 확인 필요: Ollama API의 정확한 요청/응답 스키마(`/api/chat` vs `/api/generate`, 파라미터명 등)는 설치된 Ollama 버전에 따라 다를 수 있어, 실제 구현 시 로컬 환경에서 직접 검증이 필요합니다.

## 4. 분석 엔진(서버 내부) 모듈 구조

원본과 **동일** (AI와 무관한 영역이므로 변경 없음):

1. **CSV 파서 모듈**: 파일 로드 → 행렬 변환, 타입 감지(숫자/문자열/날짜/**불리언**), 샘플링 옵션
   > ⚠️ 불리언 표기 변형 주의: 실습 데이터셋 중 `Churn` 컬럼처럼 `"True."`, `"False."`(마침표 포함 문자열) 형태로 들어오는 경우가 확인됨. 단순 `=== "true"` 비교로는 매칭 실패하므로, 타입 감지 로직에 `"true"/"True"/"True."/"TRUE"` 등 변형을 정규화하는 전처리(trim + 소문자화 + 마침표 제거 후 비교)를 포함할 것. 정규화 실패 시 해당 컬럼이 범주형 대신 일반 문자열로 분류되어 빈도 분석(모듈 2)이 누락될 수 있음.
2. **요약 통계 모듈**: 수치형(평균/중앙값/분산/표준편차/최소·최대/사분위수), 범주형(유니크 값·빈도)
3. **결측치 분석 모듈**: 컬럼별 결측치 개수/비율, 행 기준 결측 패턴
4. **상관분석 모듈**: Pearson/Spearman 상관계수 계산, 상관행렬 반환
5. **보고서 생성 모듈**: EDA 결과를 프롬프트로 포맷팅 → **EXAONE 응답**을 받아 Markdown으로 구조화

## 5. 데이터 흐름 상세

### 5.1 파일 업로드 단계 (원본과 동일)
```
사용자 파일 선택 → FileReader API → Papaparse.parse() → JSON 변환
→ State 저장(rawData, columns, rows) → 데이터 품질 체크 → 테이블 화면 이동
```

### 5.2 EXAONE 분석 요청 단계 (변경)
```
사용자 "인사이트 요청" 버튼 클릭
  → 데이터 샘플링 (최대 100행)
  → 프롬프트 생성:
     "다음 CSV 데이터를 분석하고 주요 인사이트를 한국어로 제공해주세요:
      [컬럼 정보] / [샘플 데이터 5행] / [기본 통계]"
  → 백엔드 프록시 경유 → Ollama API 호출 (model: exaone3.5:7.8b)
  → 응답 수신 (스트리밍 여부는 Ollama 설정에 따름)
  → 마크다운 형식으로 결과 렌더링
```

> ⚠️ 샘플링-통계 괴리 주의: `/api/analyze`의 요약 통계(평균·상관계수 등)는 **전체 N행 기준**으로 계산되지만, EXAONE에게 전달되는 원본 데이터는 최대 100행 샘플입니다. 데이터셋이 100행을 초과하는 경우(예: Wine Quality 4,898행), 이 둘을 명확히 구분하지 않으면 EXAONE이 마치 전체 데이터를 직접 본 것처럼 서술할 위험이 있습니다.
> → 프롬프트 템플릿에 다음을 명시할 것: "아래 통계는 전체 {N}행 기준으로 계산되었으며, 첨부된 샘플 데이터는 그 중 일부({sample_n}행)입니다. 인사이트는 통계 수치에 근거해 작성하고, 샘플 데이터로 전체를 일반화하지 마세요."
> → 샘플링 방식(상위 N행 vs 무작위 추출)도 구현 시 결정 필요(섹션 8에 추가).

### 5.3 차트 생성 단계 (원본과 동일)
```
컬럼 선택 → 데이터 타입 확인 → 차트 타입 자동 매칭 → Recharts 렌더링
```

## 6. 기술 스택

### Frontend
- Framework: React 18 + Vite
- 언어: TypeScript
- 스타일링: Tailwind CSS
- 상태 관리: Zustand
- 라우팅: React Router v6

### 데이터 처리
- CSV 파싱: Papaparse
- 차트: Recharts
- 데이터 조작: Lodash

### AI Integration (변경 지점)
- **엔진**: Ollama (로컬 LLM 런타임)
- **모델**: `exaone3.5:7.8b`
- **API 엔드포인트**: `http://localhost:11434/api/chat`
- **백엔드 프록시**: Node/Express 또는 Python/FastAPI (CORS·로컬호스트 통신 처리용)

### 배포
- 호스팅: Vercel/Netlify (프론트) — 단, **EXAONE 백엔드는 로컬/사내망에서만 구동 가능**하므로 외부 배포 시 별도 서버 구성 필요 (확인 필요 사항)

## 7. 보안 및 프라이버시 (원본 대비 강화)

| 항목 | 원본(GPT) | 변경(EXAONE) |
|---|---|---|
| API 키 관리 | 환경 변수 필수, 노출 주의 | **불필요** |
| 데이터 외부 전송 | GPT API 호출 시 샘플 데이터 전송 | **없음** (전 과정 로컬) |
| 외부 정책 의존 | OpenAI 데이터 처리 정책 적용 | **해당 없음** |
| 공공/민감 데이터 적합성 | 제한적 (외부 전송 우려) | **적합** (로컬 처리로 보안 요건 충족 가능성↑) |

> 참고: 공공기관 데이터의 경우 해싱·마스킹·익명화 처리를 거치면 외부 클라우드 도구도 법적으로 활용 가능하나, 로컬 LLM은 이 전처리 단계 자체를 생략할 수 있다는 점에서 구조적으로 유리합니다.
